## The Five Layers

Every check-in attempt passes through five independent security checks. All five must pass.

```
POST /api/checkin
       │
       ▼
1. Token valid? ──NO──▶ 404 TOKEN_NOT_FOUND
       │
       ▼
2. Token claimed? ─NO──▶ 400 TOKEN_NOT_CLAIMED
       │
       ▼
3. Token expired? ─YES─▶ 410 TOKEN_EXPIRED
       │
       ▼
4. In geofence? ───NO──▶ 403 OUTSIDE_GEOFENCE
       │
       ▼
5. Device dup? ────YES─▶ 409 DEVICE_ALREADY_CHECKED_IN
       │
       ▼
     ✓ CHECK-IN RECORDED
```

---

## Layer 1 — Token Existence

The token in the URL must exist in the database. Since tokens are randomly generated UUIDs, they can't be guessed.

**What it prevents:** Someone constructing a fake check-in URL from scratch.

---

## Layer 2 — Token Must Be Claimed First

The token must have been claimed (scanned from the kiosk). The claim step happens when the `/checkin` page loads — not at submission time. If someone somehow got a token value but didn't scan it from the kiosk, the submission fails.

**What it prevents:** Someone finding out a token value (e.g., from network traffic) and submitting directly without scanning.

---

## Layer 3 — Token Expiry

Even a claimed token expires 10 minutes after it was claimed. This limits the window of opportunity for any attack.

```ts
const claimedAt = token.claimedAt ?? token.createdAt;
if (Date.now() - claimedAt.getTime() > SUBMISSION_TTL_MS) {
  return NextResponse.json({ error: "TOKEN_EXPIRED" }, { status: 410 });
}
```

**What it prevents:** Someone saving a claimed token URL and using it hours/days later.

---

## Layer 4 — Geofencing

The user's GPS coordinates must be within `radius` meters of at least one active GeoLocation.

**What it prevents:** Remote check-ins — someone checking in from home by following a shared link.

→ See [[06 - Geofencing]] for the full implementation.

**Can be bypassed by admin** → See [[08 - Admin Dashboard#Toggle Switches]]

---

## Layer 5 — Device Deduplication {#Device Deduplication}

The same device can't check in twice on the same day.

```ts
const startOfDay = new Date();
startOfDay.setHours(0, 0, 0, 0);

const duplicate = await prisma.checkIn.findFirst({
  where: { deviceId, submittedAt: { gte: startOfDay } },
});

if (duplicate) {
  return NextResponse.json({ error: "DEVICE_ALREADY_CHECKED_IN" }, { status: 409 });
}
```

**Device ID** is generated by FingerprintJS — a hash based on browser, OS, screen, and hardware characteristics. It's stable (same value on every visit) but not tied to any user account.

**What it prevents:** One person checking in, then letting friends use their phone to check in too. Or checking in twice with the same device.

**Limitation:** FingerprintJS isn't perfect. Two very similar devices (same phone model, OS version, screen resolution) might share a fingerprint. It's a strong signal, not a guarantee.

**Can be bypassed by admin** → See [[08 - Admin Dashboard#Toggle Switches]]

## Admin Security

The admin section is protected by a single password stored as an environment variable (`ADMIN_PASSWORD`). It's compared against a cookie value set at login.

**The cookie properties:**
- `HttpOnly` — JavaScript on the page can't read the cookie value. Protects against XSS attacks where a malicious script tries to steal session data.
- `SameSite=Lax` — Browser won't send the cookie on cross-site requests (like a link from another site). Protects against CSRF attacks.
- `Secure` (in production) — Cookie only travels over HTTPS. Prevents interception on the network.
- `Path=/` — Cookie sent on all requests to this domain.
- Expires in 8 hours.

**What admin auth does NOT protect against:**
- Someone with the password
- Brute force (no rate limiting on `/api/admin/login`)

For a more hardened system, you'd add rate limiting and possibly a second factor. For an internal church tool, this is appropriate.

## The Atomic Transaction

The most critical operation — marking a token used and creating a check-in — happens in a **database transaction**:

```ts
await prisma.$transaction([
  prisma.checkIn.create({ data: { ... } }),
  prisma.token.update({ where: { id: token.id }, data: { used: true, usedAt: new Date() } }),
]);
```

A transaction means: either both operations succeed, or neither does. This prevents a race condition where:
1. Check-in is created
2. Server crashes before marking token as used
3. Same token is used again → duplicate check-in

With a transaction, if step 2 fails, step 1 is rolled back.

---

## Data Stored for Audit

Every check-in records:
- `ipAddress` — The IP address of the request (via `x-forwarded-for` header from Vercel)
- `userAgent` — The browser/OS string
- `deviceId` — FingerprintJS hash
- `latitude/longitude` — GPS at time of check-in
- `submittedAt` — Exact timestamp

This creates a detailed audit trail if there's ever a dispute about whether someone attended.
